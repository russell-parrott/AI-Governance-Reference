# GF-14 — Governance Lost in System Integration

> **Multiple components and vendors combine to create a system no single party fully governs.**

---

## What This Failure Is

The deployed AI system is not a single thing. It is an assembly. A foundation model from one vendor. A retrieval layer from another. A fine-tuning process conducted internally. An API wrapper built by a third party. A scoring component provided under a data partnership. A deployment infrastructure operated by a cloud provider. Each component has its own governance arrangements. No single party has governance visibility of the whole.

The governed components do not add up to a governed system. Governance of each part is not governance of their combination.

---

## Where It Occurs

This failure is most acute in AI deployments that rely on multiple large language model providers, data vendors and integration partners; in systems assembled from commercially available components rather than built end to end; in regulated organisations that procure AI capability from outside their regulated perimeter; and in any deployment where the organisation responsible to regulators is not the same as the organisation with the deepest technical understanding of the system.

---

## Why It Matters

Regulators hold deploying organisations accountable for their systems — not for each component individually. The organisation that deployed the integrated system is the organisation that answers for its governance. When that organisation cannot describe how the system works in combination, cannot identify who is responsible for which component's governance, and cannot produce a unified governance record for the integrated system, it cannot demonstrate adequate governance — regardless of how well each component is individually governed.

---

## How It Is Discovered

This failure is discovered when investigators ask questions about the integrated system and the organisation can only answer component by component; when an incident cannot be attributed to a single component because it arises from the interaction of multiple components; and when governance documentation describes each component's individual governance without addressing the governance of their combination.

---

## Evidence That Reveals It

Governance documentation that covers components individually without addressing integrated system governance; absence of a system-level risk assessment covering component interactions; inability of any single party to describe the full decision logic of the integrated system; and vendor contracts that define individual component obligations without establishing governance of the combined deployment.

---

## Governance Concepts Involved

→ SP-25 (System Architecture) — the structural context within which this failure occurs
→ AR-08 (Vendor Responsibility) — the accountability boundary that is unclear in multi-vendor systems
→ AR-12 (Responsibility Gaps) — the accountability vacuum at integration points
→ RE-13 (System Documentation) — the documentation that must cover the integrated system
→ RE-14 (Traceability) — the capability that integration complexity most threatens

---

## What Organisations Often Assume Instead

That governing each component satisfies the obligation to govern the integrated system. That vendor contracts for each component together constitute governance of the whole. That a system assembled from individually compliant components is itself compliant.
